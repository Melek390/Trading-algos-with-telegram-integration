[6] ALGO_002 Close All - multiple bugs fixed across several iterations (FIXED in backtesting)
---------------------------------------------------------------------------------------------
Original symptoms:
  A) "No open positions found" even though ANGO/HTH were visible in Alpaca portfolio
  B) Errors shown multiple times per symbol (one per step instead of combined)
  C) UUID not subscriptable crash (o.id[:8] on a UUID object)
  D) "pending cancel" error caused the entire position close to abort
  E) Orphaned Alpaca positions (no DB record) were never closed by Close All
  F) No market hours check -- orders submitted when market was closed (rejected silently)

Root causes and fixes applied:
  Bug A (DB-only iteration):
    Old code iterated db_positions.items() -- only positions tracked in the DB.
    If DB was cleared manually or a position was never recorded, Close All was blind to it.
    Fix: now iterates union of alpaca_map.keys() | db_positions.keys() -- closes ALL Alpaca
    positions regardless of whether a DB record exists.

  Bug B (duplicate errors):
    Each step (cancel, close, DB) appended to errors independently -> same symbol appeared
    multiple times in the error list.
    Fix: collect all step errors into sym_errors list per symbol, join with " | ", append once.

  Bug C (UUID crash):
    o.id is a UUID object in the Alpaca SDK -- slicing o.id[:8] raised TypeError.
    Fix: str(o.id)[:8] wherever order IDs are displayed.

  Bug D (pending_cancel aborts close):
    cancel_order_by_id() raises if order is already in "pending_cancel" state.
    Old code: one try/except wrapped cancel + close + DB update -> cancel failure aborted all.
    Fix: 3 independent try/except blocks. If cancel fails with "pending cancel", set
    orders_pending_cancel=True and SKIP close_position() (Alpaca holds qty until cancel
    completes -- close would fail with "insufficient qty"). DB is always updated.

  Bug E (orphaned positions):
    Alpaca positions with no matching DB record (e.g. entered via API, DB was reset) were
    invisible to Close All. "No open positions found" was shown even though Alpaca held them.
    Fix: iterate all Alpaca positions first (see Bug A fix above).

  Bug F (no market hours check):
    Submitting a close order when market is closed would fail silently or error.
    Fix: _close_all_sync() calls client.get_clock().is_open for equity algos (001/002).
    If market is closed, returns {"market_closed": True} immediately -- no orders submitted.
    Bot displays: "Market is closed. Positions can only be closed during trading hours."
    Crypto (ALGO_003) is 24/7 -- no clock check needed there.

How Close All works now (algo_002):
  1. get_clock().is_open -> if False, abort with market_closed message
  2. Fetch all Alpaca positions + all open orders + all DB open records
  3. For each symbol in (Alpaca positions union DB positions):
       a. Cancel all open orders for that symbol
          -> if "pending cancel" error: skip Alpaca close (Alpaca will auto-close when done)
       b. Close Alpaca position with market order (skipped if pending_cancel)
       c. Mark DB record closed at current price with reason "manual_close" (if record exists)
       d. Append symbol to closed list regardless
  4. Display: Closed (N): SYM1  SYM2  ...
              Errors: sym: reason (only if any step failed)
              "No open positions found" only if both Alpaca AND DB had zero open records

Files   : telegram/services/trade.py -> _close_all_sync()
          telegram/handlers/callbacks.py -> close_all_confirm handler

[8] ALGO_002 - Bracket TP rejected for penny stocks (FIXED in backtesting)
---------------------------------------------------------------------------
Symptom : Entry fails with Alpaca error:
            "takeprofit.limitprice must be >= baseprice + 0.01"
          Seen on PRPL at $0.6905 -- stock enters candidate list but order is rejected.
Cause   : tp_price = round(last_price * 1.02, 2)
          For cheap stocks, rounding to 2 decimals produces a TP that is less than
          $0.01 above the entry price, violating Alpaca's minimum bracket spread.
          Example: round(0.6905 * 1.02, 2) = $0.70 -> spread = $0.0095 < $0.01 required.
Fix     : After calculating tp_price, clamp it up if too close:
            if tp_price < last_price + 0.01:
                tp_price = round(last_price + 0.01, 2)
          This satisfies Alpaca's minimum while keeping the normal 2% TP for all other stocks.
Note    : ALEX error ("asset not active") in the same screenshot is NOT a bug --
          ALEX is simply not listed as a tradeable asset on Alpaca paper trading.
File    : portfolio_manager/trader/algo002_trader.py -> bracket order block

[9] ALGO_002 - Wrong exit data in DB + race condition + missing TP/SL notifications (FIXED in backtesting)
----------------------------------------------------------------------------------------------------------
Symptoms:
  A) HTH recorded exit_price=$36.07 reason="bracket_exit" pnl=+1.46% but Alpaca shows
     actual TP fill at $36.26 type=limit -> should be "take_profit" pnl=+1.94%
  B) NAVN hit TP while bot was live but no Telegram notification was sent
  C) Monitoring cycle and TradingStream were both closing the same position -- second
     write silently overwrote the first with wrong data (race condition)
  D) All position prices (entry and exit) were sourced from yfinance, not Alpaca

Root causes and fixes:

  Bug A - Stale order matching in _get_bracket_exit():
    Old: queried 10 closed orders for the symbol with no date filter -> matched old
         orders from a previous position in the same symbol (e.g. HTH traded twice).
    Fix: added entry_date filter -- only considers sell orders where
         filled_at.date() >= entry_date of the current DB position.

  Bug B - No real-time TP/SL notification existed:
    NAVN and HTH were missed because TradingStream (trade_stream.py) did not exist yet
    when they closed. Old code had no live bracket fill listener at all.
    Fix: TradingStream added -- subscribes to Alpaca WebSocket on bot startup.
         Fires instantly on SELL fill (limit=take_profit, stop=stop_loss):
         closes DB record + sends Telegram push notification immediately.
    Note: NAVN/HTH closed before TradingStream was deployed -- not reproducible going forward.

  Bug C - Race condition: double-write overwrites correct data:
    TradingStream closes position with correct data, then monitoring cycle runs later
    and calls close_position() again with potentially stale/wrong data -- second UPDATE wins.
    Old: close_position() did UPDATE WHERE id=? with no status check -> always overwrote.
    Fix: close_position() now checks status first -- only proceeds if status='open'.
         First caller (stream or monitoring cycle) wins. Second call is a no-op.
         Eliminates all race conditions between the two close paths.

  Bug D - yfinance used for all position prices:
    Old entry_price: yfinance fast_info.last_price at order submission time.
    Old exit_price fallback: yfinance current price if no Alpaca order matched.
    Fix entry: _get_last_price() now uses Alpaca StockLatestTradeRequest (data client).
               TradingStream buy fill event calls update_entry_price_002(order_id, fill)
               to overwrite with exact Alpaca filled_avg_price.
    Fix exit:  yfinance fallback removed from _get_bracket_exit() and run_monitoring_cycle().
               If no Alpaca fill found, DB records None -- never a fabricated price.

  Cleanup - hourly DB sync removed:
    The hourly _db_sync_loop was a third close path competing with TradingStream and
    monitoring cycle, contributing to the race condition and potential duplicates.
    Removed entirely. TradingStream handles real-time exits. Monitoring cycle in execute()
    handles time_exit (21-day) detection and catches missed exits from the trade cycle.

[14] ALGO_002 - Entry skips unavailable Alpaca assets and tries next qualified stock (FIXED in backtesting)
-----------------------------------------------------------------------------------------------------------

          seen for ALEX, PRPL on paper trading), execute() reported an error and entered
          nothing -- even though lower-ranked qualified stocks were available.

          the loop. Max 1 entry per cycle is still enforced (bug #12 unchanged).
          result.qualified is set to all_qualified[idx+1:] after success or non-avail error,
          so watchlist buttons always reflect stocks that were not entered this cycle.
